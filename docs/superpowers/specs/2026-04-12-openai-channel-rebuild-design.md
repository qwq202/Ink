# OpenAI 渠道重构设计

## 背景

当前项目里的 `OpenAI` 渠道只是一条简化过的图片生成通道，主要问题有：

- 代码把不同 OpenAI 图片能力混在一起处理，后续继续加功能会越来越乱
- 当前实现更偏旧接口思路，和最新官方文档不完全一致
- `txt2img` 和 `img2img` 在 OpenAI 渠道下没有形成清晰、完整、可扩展的能力边界
- 项目已经存在多供应商结构，如果继续在原逻辑上打补丁，会让 OpenAI 渠道越来越难维护

这次不做最小修正，目标是把 `OpenAI` 渠道做成一次真正的重构。

## 目标

- 保留一个统一的 `OpenAI` 供应商入口，不拆成多个供应商
- 在 `OpenAI` 渠道下同时支持两种官方工作方式：
  - `Image API`
  - `Responses API`
- 让用户可以显式选择当前使用哪种接口模式
- 让两种模式在代码里真正隔离，而不是继续塞进一个大函数里
- 保持现有结果区、历史记录、下载、队列等公共能力继续可用

## 非目标

- 不改 `NewAPI` 渠道。它继续作为独立供应商存在
- 不重构 `FAL`、`OpenRouter`、`Gemini` 渠道
- 不在这次设计里顺手统一全项目所有供应商协议层

## 用户侧产品形态

用户在页面中仍然只会看到一个 `OpenAI` 供应商。

当用户选择 `OpenAI` 后，表单中新增一个明确的接口模式切换：

- `Image API`
- `Responses API`

这个切换是显式的，不做自动判断，不藏到高级设置里。原因很简单：

- 两套接口的能力边界不同
- 两套接口的心智模型不同
- 用户主动选择，能减少“为什么这次是这样生成”的困惑

## 大白话工作流

### 1. Image API 模式

这是传统出图模式。

- 没上传图片时，走文生图
- 上传图片后，走图片编辑
- 更像“发起一次任务，拿回一批图片”

适合：

- 快速出图
- 单次编辑
- 明确参数控制

### 2. Responses API 模式

这是会话式出图模式。

- 第一轮可以直接生成图片
- 后续可以基于上一轮结果继续改
- 更像“围绕一张图持续对话和迭代”

适合：

- 连续追改
- 多轮创作
- 需要保留上下文的图片工作流

## 架构设计

OpenAI 渠道拆成四层：

### 1. OpenAI 渠道适配层

职责：

- 接住表单传入的 OpenAI 请求
- 根据 `apiMode` 决定走哪套引擎
- 校验最基础的必填项

它不负责拼接具体 OpenAI 请求体，也不负责解析原始响应。

### 2. Image API 引擎

职责：

- 处理 `Image API` 模式的所有请求
- 无图时走图片生成
- 有图时走图片编辑
- 只关心 Image API 自己的参数、限制和返回结构

### 3. Responses API 引擎

职责：

- 处理 `Responses API` 模式的所有请求
- 负责组织会话输入
- 负责组织图片生成工具相关参数
- 负责解析会话式图片输出

### 4. OpenAI 结果标准化层

职责：

- 把两种引擎的输出统一成项目内部格式
- 让结果展示、历史记录、下载、任务队列继续复用现有公共能力

这层的存在非常关键。这样公共模块不需要知道底层到底是 `Image API` 还是 `Responses API`。

## 数据模型调整

### OpenAI 供应商配置新增字段

- `apiMode: "image" | "responses"`

用途：

- 记录 OpenAI 当前默认接口模式
- 支持用户保存配置后，下次打开仍保持之前的选择

### OpenAI 历史记录新增信息

OpenAI 历史需要补充以下元数据：

- `openaiApiMode`
- `openaiModel`
- `openaiOperation`
- `sourceResponseId`（仅 Responses API 需要）
- 规范化后的参数快照

其中：

- `openaiOperation` 用于标识这次是生成、编辑、还是基于上一轮继续改
- `sourceResponseId` 用于支持 Responses API 的后续迭代

## UI 设计

### OpenAI 供应商下的新增元素

当供应商是 `OpenAI` 时，新增一个接口模式切换组件。

显示顺序建议：

1. 供应商选择
2. OpenAI 接口模式选择
3. 当前模式对应的模型选择
4. 当前模式对应的参数区域

### Image API 模式下的表单

保留“单次任务”心智，但只展示真实支持的参数。

核心区域：

- 模型
- 尺寸
- 质量
- 背景
- 输出格式
- 压缩率（仅 jpeg/webp）

规则：

- `txt2img` 时走生成
- `img2img` 时走编辑
- 不展示 Responses 专属上下文信息

### Responses API 模式下的表单

强调“会话式”体验。

核心区域：

- 主模型
- 图片生成工具相关配置
- 当前轮输入
- 上一轮上下文提示

规则：

- 支持从上一轮继续改图
- 支持在历史中回到某一轮继续创作
- 不强行复用 Image API 的所有参数展示

## 请求流设计

### 统一入口

`GenerationForm` 仍然只负责收集用户输入。

`useGeneration` 仍然负责：

- 队列
- 重试
- toast
- 历史写入

OpenAI 协议细节全部下沉到 OpenAI 专用适配层和引擎。

### 路由规则

OpenAI 适配层按以下逻辑分发：

- `apiMode = image` 且无上传图：走图片生成
- `apiMode = image` 且有上传图：走图片编辑
- `apiMode = responses`：走 Responses API 引擎

## 错误处理

OpenAI 渠道错误分三类：

### 1. 配置错误

例如：

- 没填 API Key
- 没选接口模式
- 没选模型

这类错误要在前端尽量提前拦住，给出明确提示，不要让请求打到服务端后再失败。

### 2. 参数错误

例如：

- 当前模式不支持某参数
- 图片编辑缺必要输入
- 输出格式和背景参数冲突

这类错误要在 OpenAI 适配层统一校验。

### 3. 上游错误

例如：

- OpenAI 返回 4xx/5xx
- 限流
- 响应结构异常

这类错误由具体引擎负责转换成项目内统一错误消息。

## 兼容策略

这次重构采用软迁移，不做激进破坏。

### 配置兼容

旧的 OpenAI 配置自动补默认值：

- `apiMode` 默认补成 `image`

这样老用户升级后不会突然失效。

### 历史兼容

旧历史记录继续可读，不强制整体迁移旧数据。

新生成的 OpenAI 记录再写入新增字段即可。

### 公共模块兼容

结果区、下载、历史列表、任务队列继续使用统一内部结果格式，不需要针对 OpenAI 渠道大改。

## 实施范围

本次重构至少会涉及：

- OpenAI 供应商配置定义
- OpenAI 设置面板
- 生成表单中 OpenAI 相关 UI
- OpenAI 请求执行层
- OpenAI 响应解析层
- OpenAI 历史记录元数据
- OpenAI 渠道的回填和重新生成逻辑

## 验收标准

满足以下条件即可视为这次重构完成：

- 用户选择 `OpenAI` 后，可以明确切换 `Image API` 和 `Responses API`
- `Image API` 模式下，文生图和图片编辑都能正常工作
- `Responses API` 模式下，可以完成首轮生成和基于上一轮继续编辑
- 两种模式的结果都能正常展示、下载和写入历史
- 历史回填时能够恢复正确的 OpenAI 模式和关键参数
- 不影响 `NewAPI`、`FAL`、`OpenRouter`、`Gemini` 原有渠道

## 风险与控制

### 风险 1：OpenAI 参数差异导致表单变复杂

控制方式：

- 明确区分两种模式的参数面板
- 不追求“所有参数全复用”

### 风险 2：Responses API 上下文逻辑影响现有历史体系

控制方式：

- 只给 OpenAI 历史补最少但足够的上下文字段
- 不把整个历史系统改成会话数据库

### 风险 3：继续把逻辑塞回单个文件

控制方式：

- 从设计上要求拆成适配层、双引擎、标准化层
- 禁止把两套协议重新并回一个超大函数

## 推荐实现顺序

1. 先扩展 OpenAI 配置模型，加入 `apiMode`
2. 再完成 OpenAI UI 切换与参数面板拆分
3. 接着拆分 OpenAI 执行层为双引擎
4. 最后补历史记录、回填、继续编辑链路

## 最终结论

这次重构的本质不是“给 OpenAI 渠道补几个参数”，而是把它从一条混杂逻辑的旧通道，升级成一个：

- 对用户仍然简单
- 对代码内部真正解耦
- 同时支持 `Image API` 与 `Responses API`
- 能继续演进的正式渠道

这是一次面向后续维护和扩展的结构性重建。
